第二阶段：思考 → 问题与优化
🔴 一、架构层问题
1. 双 UI 体系 = 双份维护成本
事实：fluent_app_shell.dart 和 material_app_shell.dart 维护两套几乎相同的页面，只是控件名称不同。NavigationView vs NavigationRail，PaneItem vs NavigationDestination。
Linus 的判断：这是懒惰的分歧。核心业务逻辑（图库、搜索、标签管理、设置）与 UI 外壳无关。两套实现意味着改一个功能要动两个文件，漏改一处就是 bug。
优化方案：
lib/
├── ui/
│   ├── components/              ← 平台无关的通用组件
│   │   ├── image_grid.dart
│   │   ├── tag_chip.dart
│   │   ├── search_bar.dart
│   │   └── metadata_panel.dart   ← 元数据面板已部分抽象
│   ├── fluent/                  ← Fluent 特有包装（只做风格适配）
│   │   ├── navigation_adapter.dart
│   │   └── theme_bridge.dart
│   ├── material/                ← Material 特有包装
│   │   ├── navigation_adapter.dart
│   │   └── theme_bridge.dart
│   └── shell/                   ← 统一壳层，注入不同 adapter
│       └── unified_shell.dart
用 Adapter 模式抽象导航：
abstract class NavigationAdapter {
  Widget build(List<NavigationItem> items, int selectedIndex, ValueChanged<int> onChanged);
}
壳层只传入业务数据，adapter 负责渲染成 Fluent 或 Material 控件。页面内容 100% 共享。
2. 无路由系统 = 深层链接不可能
事实：项目用 NavigationProvider.selectedIndex 索引切换页面，没有 go_router。
后果：
- 无法直接从 URL 跳转到搜索页并带查询参数
- 无法分享某个搜索结果或特定图片的链接
- Web 端刷新后状态丢失
- 无法做浏览器前进/后退
Linus 的建议：
// go_router 配置示例
final router = GoRouter(
  routes: [
    GoRoute(path: '/', builder: (_, __) => GalleryPage()),
    GoRoute(path: '/search', builder: (_, state) => SearchPage(query: state.uri.queryParameters['q'])),
    GoRoute(path: '/tags', builder: (_, __) => TagManagementPage()),
    GoRoute(path: '/settings', builder: (_, __) => SettingsPage()),
    GoRoute(path: '/viewer/:imageId', builder: (_, state) => ViewerWindowService.openWindow(int.parse(state.pathParameters['imageId']!'))),
  ],
);
这不是过度工程。Web 端没有路由就是一个半残产品。
3. Viewer 窗口数据加载策略低效
事实：viewer_workspace.dart 中 _handleNext() / _handlePrevious() 每次切换都调用 _loadWindow(nextContext)，即发一次完整 API 请求获取整个 window 数据。
Future<void> _loadWindow(ViewerWindowContext context) async {
  final window = await widget.apiService.fetchViewerWindow(context);  // 每次都请求 API
  ...
}
问题：切换 10 张图片 = 10 次 API 调用。用户连续按键盘浏览时，大量串行请求排队。
优化方案：
- 预加载：当前显示第 3 张时，后台预加载第 4、5 张的数据
- 本地缓存：已获取的 window 数据缓存在内存中，翻页不重复请求
- 批量窗口：一次请求更大的 window（如一次 50 张），客户端本地切换，超出边界再请求
class ViewerWindowCache {
  final Map<int, ViewerWindowResult> _cache = {};
  int _windowStartIndex = 0;
  static const int windowSize = 50;
  ViewerWindowResult? getAt(int index) {
    if (index < _windowStartIndex || index >= _windowStartIndex + windowSize) return null;
    return _cache[index - _windowStartIndex];
  }
}

🟡 二、功能层缺失
4. 图片操作能力严重不足
当前 viewer 只能看和管理标签，但一个真正的图片管理工具应该能：
缺失功能	价值
原图下载/导出	用户看到好图需要保存
收藏/评分	快速标记好图，后续检索
批量操作	多张图片统一加标签/移动
对比模式	左右对比两张相似图，决定去重
全屏查看	F11 无边框展示
旋转/翻转	图片方向修正
复制图片路径	右键菜单复制文件系统路径
在文件管理器中打开	直接打开图片所在文件夹
Linus 的优先级排序：先做 复制路径 + 文件管理器打开 + 全屏，这三个成本最低但用户感知最强。
5. 搜索只有基础标签过滤
当前搜索：文本查询 → AND 语义 → 返回结果。
应该有的：
高级搜索
日期范围
文件大小
分辨率范围
标签组合
排除标签
相似度搜索
保存搜索条件
6. 图库浏览无虚拟滚动/懒加载
事实：gallery 页面如果管理数万张图片，全部渲染会撑爆内存。
检查项：需要确认 image_gallery_viewer.dart 是否使用了 ListView.builder / SliverGrid 做了虚拟化。如果用了 ListView + 静态 children，那就是一个 严重性能 bug。

🟡 三、工程质量
7. TagChip 交互逻辑重复
事实：ImageMetadataPanel 中 _confirmTag, _rejectTag, _removeTag, _showEditTagDialog, _showMergeDialog, _addTag 六个方法全部直接写在 widget 里，每个都重复：
1. 获取 TagProvider
2. 调用 provider 方法
3. catch 异常
4. showSnackBar
这是典型的应该提取的东西：
mixin TagActionsMixin<T extends StatefulWidget> on State<T> {
  TagProvider get tagProvider => context.read<TagProvider>();
  Future<void> confirmTag(int imageId, int tagId) async {
    try {
      await tagProvider.confirmImageTag(imageId, tagId);
    } catch (e) {
      if (mounted) _showError('确认标签失败: $e');
    }
  }
  // ... 统一错误处理
}
8. 硬编码的 API 地址
事实：viewer_window_app.dart 第 58 行：
apiService ?? ApiService(baseUrl: 'http://localhost:8080')
问题：如果后端不在 localhost 或者端口改了，viewer 窗口直接连不上。主窗口用 ConfigProvider.baseUrl，但 viewer 的兜底地址硬编码。
修复：viewer 窗口启动参数应该包含 baseUrl。
9. 空 catch 吞错误
事实：多处 catch (_) { } 不做任何处理。
// viewer_window_service.dart
for (final id in subWindowIds) {
  try {
    await WindowController.fromWindowId(id).close().timeout(...);
  } catch (_) {
    // Best-effort close only.
  }
}
Linus 的态度：catch (_) {} 是 代码异味。至少在 debug 模式下应该 debugPrint 日志。生产环境可以静默，但开发时要知道发生了什么。

## 第四阶段：测试 → 验证要点
对上述改动，需要验证：
1. **baseUrl 修复**：viewer 窗口是否能正确连接非 localhost 后端
2. **预加载**：快速连续按 → 键盘 → 切换图片时，无加载菊花闪烁
3. **路由系统**：刷新 Web 页面，URL 状态保持；直接访问 `/search?q=blue_hair` 正确加载
4. **虚拟滚动**：加载 10000 张图片的图库 → 内存不超 200MB
