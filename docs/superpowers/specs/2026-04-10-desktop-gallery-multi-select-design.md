# Windows 桌面图库多选与批量操作设计

## 背景

当前 Windows 桌面端 Fluent 图库页已经存在部分批量能力的按钮定义，但用户在实际界面里没有明确的“进入多选模式”入口，因此无法顺畅完成多选图片后的批量操作。现有状态会让界面看起来“似乎支持批量操作”，但关键入口缺失，导致能力不可达。

本设计的目标是在不打乱当前桌面图库主流程的前提下，为 Fluent 桌面路径补齐一套明确、可发现、符合桌面习惯的多选交互。

## 目标

1. 用户可以从图库工具栏显式进入选择模式。
2. 用户可以逐张选中图片，并执行全选、全不选。
3. 用户可以对已选图片执行批量添加标签与批量删除。
4. 现有“批量AI标签”能力保持原有语义和入口，不因多选设计而被改成“仅针对已选图片”。
5. 所有交互都落在当前 Windows Fluent 桌面图库路径中，不依赖 Material 路径的长按选择逻辑。

## 非目标

1. 不引入 Ctrl/Shift 桌面快捷键多选。
2. 不实现跨分页 / 全库范围的“全选全部结果”。
3. 不改造现有“批量AI标签”为按已选图片执行。
4. 不重做右键菜单逻辑。
5. 不重构现有 provider 架构，只做 Fluent 路径所需的最小补线。

## 交互设计

### 默认浏览态

图库页保持现有浏览行为：点击图片打开详情，右键打开上下文菜单，顶部仍保留现有常驻控件。

在工具栏新增一个明确的 **“选择”** 按钮，作为进入多选模式的唯一主入口。

### 选择模式

进入选择模式后：

1. 图片单击行为从“打开详情”切换为“选中 / 取消选中”。
2. 工具栏切换为选择态工具栏，而不是在原工具栏后继续追加按钮。
3. 工具栏显示当前已选数量，例如“已选 3 张”。
4. 工具栏至少提供以下动作：
   - 全不选
   - 全选
   - 批量添加标签
   - 批量删除
   - 退出选择模式

### 工具栏切换规则

为避免行为语义冲突，工具栏需要明确分成两套：

1. **浏览态工具栏**：保留当前常规动作，包括“排序”“刷新”“批量AI标签”，并新增“选择”。
2. **选择态工具栏**：只显示和选择相关的动作，即“全不选”“全选”“批量添加标签”“批量删除”“退出选择模式”。
3. 进入选择模式后，浏览态工具栏中的“排序”“刷新”“批量AI标签”“选择”不再显示。
4. 退出选择模式后，再恢复浏览态工具栏。

### 全不选 vs 退出选择模式

这两个动作必须保留为两个不同按钮，语义不能混用：

1. **全不选**：仅清空 `selectedImageIds`，但仍保留在选择模式里，方便用户重新选择。
2. **退出选择模式**：调用 `exitSelectionMode()`，同时关闭选择模式并清空当前选择。

现有 Fluent 路径里的“取消选择”按钮更接近“全不选”语义，不应继续拿它表示“退出选择模式”。实现时应统一成上述两类动作。

### 批量 AI 标签的保留策略

现有 **“批量AI标签”** 保持原有能力和语义，不并入选择模式的“已选图片批处理”语义中。

也就是说：

1. 现有页面上的“批量AI标签”按钮继续按当前实现工作。
2. 进入选择模式后，这个按钮从工具栏中隐藏，而不是改造成“仅对 selectedImageIds 生效”。
3. 本期多选设计聚焦于：选择入口、批量添加标签、批量删除、全选、全不选。

这样可以避免把一个原本按筛选条件/全局批处理工作的能力，悄悄改成另一种语义，减少用户认知变化和回归风险。

## 视觉状态

选择模式下，图片卡片需要有清晰的选中反馈：

1. 选中卡片显示高亮边框。
2. 选中卡片叠加半透明遮罩。
3. 右上角显示勾选标记。
4. 未选中卡片保持可点，但行为变为“加入选择”。
5. 选择模式下，未选中卡片右上角也应显示弱化的勾选占位或空心勾选标记，帮助用户理解当前处于“可选择”状态。

视觉目标不是重做卡片，而是在现有 `FluentImageCard` 基础上补足“已选中”状态，使用户能快速确认当前批量操作作用范围。

## 数据流设计

### SelectionProvider

继续复用现有 `SelectionProvider` 作为唯一选择态状态源。

需要明确的状态语义：

1. `isSelectionMode=true` 表示界面已进入多选模式。
2. `selectedImageIds` 保存当前已选图片集合。
3. `selectAll` 的作用范围定义为“当前前端已加载出来的图片列表”。
4. `clearSelection` 负责清空当前选中项。
5. `exitSelectionMode` 负责退出选择模式并清空选择。

### Fluent 路径补线

Fluent 桌面路径目前的主要问题不是状态对象不存在，而是没有把 Fluent 组件真正接到 `SelectionProvider` 上。需要补齐以下链路：

1. `FluentGalleryPage` 负责渲染“选择”按钮和选择态工具栏。
2. `FluentGalleryContent` 负责在构建图片网格时，把选择态和选中状态传给图片卡。
3. `FluentImageCard` 负责根据选择态切换点击行为，并渲染选中视觉状态。

## 批量操作语义

### 批量添加标签

进入选择模式后，点击“批量添加标签”时：

1. 使用当前 `selectedImageIds` 作为目标集合。
2. 复用现有批量加标签对话框/服务。
3. 成功后先 `await imageProvider.loadImages(refresh: true)`，等待列表刷新完成。
4. 刷新完成后调用 `exitSelectionMode()`，统一退出选择模式并清空选择。
5. 失败时保留当前选择状态，便于用户重试。

### 批量删除

点击“批量删除”时：

1. 必须先显示确认弹窗。
2. 明确提示将删除多少张图片。
3. 弹窗至少提供“取消”和“确认删除”两个动作，视觉上按危险操作处理。
4. 确认后调用现有批量删除服务。
5. 成功后先 `await imageProvider.loadImages(refresh: true)`，等待列表刷新完成。
6. 刷新完成后调用 `exitSelectionMode()`，统一退出选择模式并清空选择。
7. 失败时保留当前选择状态，便于用户重试或调整。
8. 若后端返回部分失败，应在错误提示中体现“部分删除成功 / 失败数量”，不要只给笼统失败文案。

### 全选 / 全不选

1. 全选：对当前 `imageProvider.images` 全量选中。
2. 全不选：清空当前选择，但保持在选择模式中。

本设计不做“跨分页全选”，避免前后端语义不清和额外接口改动。

## 错误处理

1. 批量添加标签失败时，显示明确失败提示，并保留当前选择状态，不静默吞错。
2. 批量删除失败时，保留当前选择状态，便于用户重试或调整。
3. 退出选择模式始终清空当前选择，避免旧状态残留。
4. 对零选中状态下的“批量添加标签”和“批量删除”按钮，应禁用，避免空操作。
5. 对零选中状态下的“全不选”按钮，可禁用；“全选”和“退出选择模式”仍可用。

## 受影响模块

预计影响以下模块：

- `flutter_app/lib/app/fluent_screens.dart`
- `flutter_app/lib/widgets/fluent_gallery_content.dart`
- `flutter_app/lib/widgets/fluent_image_card.dart`
- `flutter_app/lib/providers/selection_provider.dart`
- 可能新增一个 Fluent 风格的批量删除确认弹窗，或在现有页面内直接使用 `ContentDialog`

## 验收标准

1. 图库页可通过工具栏“选择”按钮进入多选模式。
2. 进入选择模式后，点击图片会选中/取消选中，而不是打开详情。
3. 用户可以执行全选和全不选。
4. 用户可以对已选图片执行批量添加标签。
5. 用户可以对已选图片执行批量删除，并经过确认弹窗。
6. 批量添加标签和批量删除成功后，图库列表正确刷新，然后退出选择模式并清空选择。
7. 批量添加标签和批量删除失败时，当前选择状态仍然保留。
8. 原有“批量AI标签”能力和语义保持不变，并仅在浏览态工具栏中可见。

## 风险与取舍

1. 选择模式引入后，同一张卡片在“浏览态”和“选择态”下有两套点击语义，必须确保视觉提示足够清楚。
2. 若把“批量AI标签”强行并入选择模式，会改变原有语义，因此本设计明确不这么做。
3. 仅做“当前已加载图片全选”是刻意取舍，虽然能力不如跨分页全选强，但能明显降低复杂度和误操作风险。

## 结论

本设计采用最小可行方案：在 Fluent 桌面图库中补上明确的选择模式入口，并把多选后的批量添加标签、批量删除、全选、全不选接通；同时明确保持现有“批量AI标签”能力不变，避免在修复多选路径时意外改变既有批处理语义。
