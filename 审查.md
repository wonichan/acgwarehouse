# Hierarchical Tag Governance — Code Review Report

**审查日期:** 2026-04-13
**审查范围:** `docs/superpowers/specs/2026-04-12-hierarchical-tag-governance-design.md` 及对应实现

---

## Summary

该实现对 Spec 的核心数据模型和大部分业务规则覆盖良好。后端功能齐全——层级字段、后代展开、运行时树统计、同级合并、变更层级、重设父级均已实现。但存在 **5 个严重问题** 和 **7 个中等问题**，应在发布前修复。

---

## 🔴 严重问题（Spec 违规 / 正确性风险）

### C1. `Save()` 使用 `INSERT OR REPLACE` — 静默数据丢失

**文件:** `internal/repository/tag_repository.go:51-53`

`Save()` 对已知 ID 的 tag 使用 `INSERT OR REPLACE`。SQLite 中这会先 `DELETE` 再 `INSERT`，导致：
- **`usage_count` 被重置为 0**（破坏 trigger 维护的计数器）
- **触发 `ON DELETE CASCADE`**，静默删除 `tag_aliases` 和 `image_tags` 的关联行
- 静默破坏层级子节点的引用

```go
result, err = r.db.ExecContext(ctx, `
    INSERT OR REPLACE INTO tags (id, preferred_label, slug, level, parent_id, ...)
    VALUES (?, ?, ?, ?, ?, ?, ?, ?, ?, ?)
```

**修复:** 对 `tag.ID > 0` 的情况改用 `UPDATE`；对新插入使用不带 `OR REPLACE` 的 `INSERT`（或用 `INSERT OR IGNORE` + `UPDATE` 做 upsert）。

---

### C2. 无环路检测

**文件:** `internal/service/tag_admin_service.go:663-706` (`validateHierarchyAssignment`)

`validateHierarchyAssignment` 检查了 `parentID == tagID`（自引用）并验证父 tag 的 level，但**从未验证新的父节点不是被操作 tag 的后代**。`ReparentTag()`（第 513 行）同样没有此检查。

恶意请求如 `reparent(child=1, parent=3)`，其中 tag 3 是 tag 1 的子节点，会形成环路。递归 CTE `resolveDescendantIDsForSingle` 会命中 SQLite 递归限制（默认 1000 次迭代）而非报错，产生不正确结果。

**修复:** 在 `ReparentTag` 和 `ChangeLevel` 中，先解析目标的祖先链，确认目标父节点不在该集合中才允许操作。

---

### C3. AI 新建 tag 未显式设置 `Level = "child"`

**文件:** `internal/service/tag_governance_service.go:87-92`

AI `MergeTags` 创建的 tag 没有设置 `Level` 和 `ParentID`：

```go
tag = &domain.Tag{
    PreferredLabel: normalized,
    Slug:           slugify(normalized),
    ReviewState:    "pending",
    TrustScore:     confidence,
}
```

Repository 的 `Save()` 会把空 `Level` 默认为 `"child"`（第 42-43 行），所以目前恰好能工作，但这是**隐式耦合**而非显式执行。如果任何人改了默认逻辑，AI 创建的 tag 就会违反 Spec 规则 4（"if not matched, create a new child tag"）。

**修复:** 显式设置 `Level: domain.TagLevelChild` 和 `ParentID: nil`。

---

### C4. 手动创建在重复匹配时静默忽略用户请求的 level

**文件:** `internal/handler/tag_hierarchy_helper.go:28-34`

当用户明确请求创建 `root` tag，但该 label 已作为 `child` 存在时，`resolveOrCreateManualTag` 会返回匹配到的现有 `child` tag 并标记 `reused = true`，**不向调用方发出任何警告**：

```go
if existing != nil {
    return existing, true, nil  // silently returns child when user asked for root
}
```

这违反了 Spec 规则 5（"manual tag creation must let the user explicitly choose root, parent, or child"）。复用行为本身是对的，但**调用方无法得知 level 不匹配**，用户的意图被静默忽略。

**修复:** 返回一个包装类型，指示是否被复用以及被复用的 tag level 是否与请求不同。让 handler/前端在 level 不匹配时展示警告。

---

### C5. `ChangeLevel` 升级时只断开直接子节点，未处理所有后代

**文件:** `internal/service/tag_admin_service.go:495-506`

当 `parent` 升级为 `root` 时，代码只将直接子节点的 `parent_id` 设为 `NULL`。但子节点的子节点（如果存在，即 3 层深度）**不会被断开**，最终会指向一个仍标记为 `child` 的 tag，而后者现在指向 `NULL`——产生一个悬挂的 2 层子树。

更关键的是，前面的检查（第 469-473 行）阻止了 `parent -> child` 在有子节点时的情况，但**没有阻止 `parent -> root` 在有后代时的情况**——它只在 `root -> non-root` 时检查 `hasDescendants`。

```go
// 只阻止 root 降级时有后代的情况
case tag.Level == domain.TagLevelRoot && targetLevel != domain.TagLevelRoot && hasDescendants:
    return nil, ErrInvalidHierarchy
```

这意味着一个有子节点的 `parent` 可以变成 `root`，而第 495-506 行的代码只断开直接子节点。孙节点成为孤儿。

**修复:** 在 `parent -> root` 晋升时也检查 `hasDescendants`，或递归重设所有后代的父级。

## 🟡 中等问题

### M1. `ListGovernanceTags` 的 N+1 查询

**文件:** `internal/service/tag_admin_service.go:130-188`

对**每个** governance 行，代码执行 3 次独立查询：`computeHierarchyStats(direct)`、`ResolveAllDescendantIDs + computeHierarchyStats(tree)`、`FindChildrenByParent`。100 个 tag 的分页会产生约 300 次查询。

**建议:** 批量解析所有后代 ID，对当前页所有 tag 执行一次统计查询。

---

### M2. `GetTagTree` 具有相同的 N+1 模式

**文件:** `internal/service/tag_admin_service.go:390-396`

对树中每个 tag 都调用 `ResolveAllDescendantIDs` 和 `computeHierarchyStats`。500 个 tag 产生约 1000 次查询。

---

### M3. Migration 和 Schema 中未设置 `CHECK(level IN ('root','parent','child'))` 约束

**文件:** `migrations/005_add_tag_hierarchy.up.sql`, `internal/repository/schema.go:35`

Schema 接受 `level` 的任意字符串值。当前仅靠应用层校验，但 SQL 层的拼写错误或绕过 handler 的直接 DB 编辑会创建无效状态。

**建议:** 在 schema.go 的 bootstrap DDL 中添加 `CHECK(level IN ('root', 'parent', 'child'))`。

---

### M4. 递归后代 CTE 包含根节点自身

**文件:** `internal/repository/tag_repository.go:254-261`

递归 CTE 的锚点成员是 `SELECT id FROM tags WHERE id = ?`，所以"后代"包含了 tag 本身。这对 `expandHierarchicalTagClauses` 的 `self + descendants` 语义是正确的，但方法名 `ResolveDescendantIDs` 和 `ResolveAllDescendantIDs` 有误导性——调用方可能期望只获取真正的后代。

**建议:** 添加注释说明"后代"包含自身，或重命名以明确语义。

---

### M5. Flutter `Tag.level` 是松散的 `String?` 而非枚举

**文件:** `flutter_app/lib/models/tag.dart`

`level` 在 Dart 中类型为 `String?`，没有枚举约束。Flutter 代码中无法在编译时检查拼写错误的 level 值。

---

### M6. Flutter 治理工作区是列表式而非树式

**文件:** `flutter_app/lib/widgets/tag_management/tag_management_workspace.dart`, `tag_management_list.dart`

治理界面将树结构扁平化为缩进列表，而非使用可展开/折叠的树控件。Spec 要求"治理 UI 必须是树感知的"并提到了"树形展示"。

---

### M7. Flutter 合并面板选择列表是扁平的

**文件:** `flutter_app/lib/widgets/tag_management/tag_merge_panel.dart`

合并目标选择是扁平列表，即使已按同级过滤。用户在选择合并目标时失去了层级上下文。

## 🟢 优点 — 做得好的地方

1. **数据模型**干净且符合 Spec：`level`、`parent_id`、迁移回填默认 `'child'`——全部正确。

2. **后代展开用于过滤**（`tag_hierarchy_query.go`）使用递归 CTE 并生成正确的 SQL AND 子句——与 Spec 的"选中 tag + 后代，AND 语义"完全一致。

3. **统计模型**正确区分了直接计数和树计数，使用 `COUNT(DISTINCT image_id)` 避免重复计数。

4. **同级合并**（`tag_admin_service.go:238`）正确阻止了跨层级合并。

5. **删除预览**（第 283-318 行）正确阻止了有子节点和有直接图片关联时的删除。

6. **Flutter 过滤面板**（`fluent_tag_filter_pane.dart`）是真正的 `TreeView`，支持展开/折叠、层级徽章、树使用计数——最符合 Spec 的 UI 元素。

7. **`validateManualTagHierarchy`**（`tag_hierarchy_helper.go`）正确执行了 parent→root、child→parent、root→无父节点的规则。
