# 技术设计：后端服务压测与并发检查

## 1. 架构概览

```
┌─────────────────────────────────────────────────────────────┐
│                    压测执行流程                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌──────────────┐    ┌──────────────┐    ┌──────────────┐  │
│  │ 数据准备阶段 │ -> │ 压测执行阶段 │ -> │ 数据清理阶段 │  │
│  └──────────────┘    └──────────────┘    └──────────────┘  │
│         │                   │                   │          │
```

## 2. 压测场景设计

### 2.1 场景矩阵

| 场景 | 端点 | 方法 | 认证 | 目的 |
|------|------|------|------|------|
| **S1: 健康检查** | `/api/v1/ping` | GET | 否 | 基础连通性 |
| **S2: 图片列表** | `/api/v1/images` | GET | 否 | 读性能基准 |
| **S3: 图片详情** | `/api/v1/images/:id` | GET | 否 | ViewBuffer 并发测试 |
| **S4: 图片搜索** | `/api/v1/search?q=tag` | GET | 否 | Bleve 并发测试 |
| **S5: 用户评分** | `/api/v1/images/:id/rating` | PUT | 是 | 写事务并发测试 |
| **S6: 收藏操作** | `/api/v1/collections` | POST | 是 | 写并发测试 |
| **S7: 热榜查询** | `/api/v1/rankings` | GET | 否 | 缓存读取性能 |
| **S8: 混合场景** | 多端点混合 | 混合 | 混合 | 真实负载模拟 |

### 2.2 并发等级设计（2c2g 约束）

| 等级 | 连接数 | 线程数 | 持续时间 | 预期 QPS |
|------|--------|--------|----------|----------|
| **L1: 低负载** | 10 | 2 | 30s | 50-100 |
| **L2: 中负载** | 50 | 4 | 60s | 200-500 |
| **L3: 高负载** | 100 | 4 | 60s | 500-1000 |
| **L4: 极限负载** | 200 | 4 | 60s | > 1000 或失败 |

**注意**：2c2g 环境下，4 线程是 wrk 的合理上限，避免测试工具本身成为瓶颈。

### 2.3 Lua 脚本设计

```lua
-- 压测脚本结构
-- 1. setup(): 登录获取 token，初始化测试数据
-- 2. init(): 每个线程初始化
-- 3. request(): 生成请求（根据场景选择端点）
-- 4. response(): 处理响应，统计错误
-- 5. done(): 汇总结果
```

**认证流程**：
1. 预创建测试用户（压测前）
2. Lua setup 阶段登录获取 JWT token
3. 认证请求携带 Authorization header

## 3. 并发问题检测策略

### 3.1 ViewBuffer 阻塞检测

**检测方法**：
- 高 QPS 图片详情请求（S3 场景）
- 监控 P99 延迟是否出现毛刺
- 检查日志中是否有 ViewBuffer 刷新错误

**判断标准**：
- P99 延迟 > 500ms 且波动大 → 存在阻塞
- 日志出现 `flush view events failed` → 数据丢失风险

### 3.2 评分一致性检测

**检测方法**：
- 并发评分同一图片（S5 场景）
- 压测后查询图片平均分
- 与理论值对比

**判断标准**：
- `avg_score` 计算错误 → 事务隔离问题
- 数据库锁超时错误 → SQLite 写竞争

### 3.3 Bleve 并发检测

**检测方法**：
- 高 QPS 搜索请求（S4 场景）
- 监控服务是否崩溃
- 检查搜索结果是否为空或异常

**判断标准**：
- 服务 panic → Bleve 并发问题
- 搜索结果异常 → 索引损坏

### 3.4 热榜任务竞争检测

**检测方法**：
- 压测期间观察热榜任务日志
- 检查热榜数据是否正常更新

**判断标准**：
- 热榜任务失败 → writeDB 竞争
- 热榜数据为空 → 写入失败

## 4. 资源监控方案

### 4.1 监控指标

| 指标 | 工具 | 采集频率 |
|------|------|----------|
| CPU 使用率 | `top` / `htop` | 1s |
| 内存使用 | `free -m` | 1s |
| 磁盘 I/O | `iostat` | 1s |
| 进程状态 | `ps aux` | 5s |
| 网络连接 | `netstat` | 5s |

### 4.2 日志收集

- 后端日志：`stdout` 重定向到文件
- wrk 输出：保存为 JSON 格式
- 错误日志：单独提取 5xx 响应

## 5. 测试数据设计

### 5.1 测试用户

| 用户名 | 密码 | 角色 | 用途 |
|--------|------|------|------|
| `loadtest_user1` | `Test123456` | user | 压测请求 |
| `loadtest_user2` | `Test123456` | user | 并发评分 |
| `loadtest_admin` | `Admin123456` | admin | 管理操作测试 |

### 5.2 测试数据范围

- **图片 ID**：使用现有图片，范围 1-1000
- **评分**：随机 0-100 分
- **收藏夹**：每个用户创建 1-3 个收藏夹
- **标签**：使用现有标签

### 5.3 数据隔离

- 所有测试数据使用 `loadtest_` 前缀
- 压测后按前缀删除，不影响生产数据

## 6. 风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 压测导致服务崩溃 | 数据丢失 | 使用独立测试数据库副本 |
| SQLite 文件损坏 | 不可恢复 | 压测前备份数据库 |
| 测试数据清理不完整 | 污染生产数据 | 使用事务，失败自动回滚 |
| 资源耗尽 | 影响其他服务 | 使用 cgroup 限制资源 |

## 7. 回滚方案

1. **压测失败**：停止 wrk，重启后端服务
2. **数据异常**：恢复数据库备份
3. **服务崩溃**：检查日志，修复后重启
