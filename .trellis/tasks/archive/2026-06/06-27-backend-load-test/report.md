## 4. 并发问题清单

### 🟡 中等问题

#### 问题 1: SQLite 高并发瓶颈

**症状**: 
- 图片详情 API 在 200 并发下出现 643 次超时
- P99 延迟达到 1.34 秒
- 图片列表 QPS 仅 191

**根因分析**:
- **SQLite 设计限制**：单文件数据库，高并发下磁盘 I/O 成为瓶颈
- 数据库文件 70MB，200 并发超出 SQLite 设计范围
- 读连接池大小 = CPU 核心数（默认配置），高并发时连接等待

**验证过程**:
1. 尝试优化 ViewBuffer 使用 channel 替代 Mutex → 超时反而增加（918 次）
2. 低并发（10 连接）测试正常：QPS 8445，P99 < 10ms
3. 单次请求响应时间正常（9ms），确认不是单个查询慢

**结论**: 这是 SQLite 本身的并发限制，而非代码问题

**影响范围**: 高并发场景（> 100 并发）下的所有数据库操作

**修复建议**:
1. **短期**：限制并发连接数 ≤ 100，使用连接池监控
2. **中期**：迁移到 PostgreSQL/MySQL（支持更高并发）
3. **长期**：添加 Redis 缓存层，减少数据库直接访问

**优先级**: 🟡 中（需要架构决策）

### 🟡 中等问题

#### 问题 3: 图片列表性能瓶颈

**症状**: 
- QPS 仅 191，远低于其他 API
- P99 延迟 820ms

**根因分析**:

**修复建议**:
1. 添加缓存层（如 Redis）
2. 使用分页优化查询
3. 考虑使用更强大的数据库（PostgreSQL）

**优先级**: 🟡 中

## 7. 结论

### 7.1 性能评估

- **健康检查**: 优秀（77K+ QPS，P99 < 10ms）
- **搜索 API**: 良好（17K+ QPS，P99 < 50ms）
- **图片详情**: ⚠️ 存在并发问题，需优化
- **图片列表**: 一般，需优化

### 7.2 并发问题总结

| 问题 | 严重程度 | 是否需要修复 |
|------|----------|--------------|
| SQLite 高并发瓶颈 | 🟡 中 | 需要架构优化（非代码 bug） |
| 连接关闭错误 | 🟢 低 | 否（正常行为） |
| 图片列表性能 | 🟡 中 | 建议优化 |

### 7.3 2c2g 环境建议

在 2c2g 环境下，建议：
1. **并发连接数不超过 100**（SQLite 并发限制）
2. 低并发场景下性能良好：健康检查 77K+ QPS，搜索 17K+ QPS
3. 如需支持更高并发，建议迁移到 PostgreSQL 或添加 Redis 缓存

### 7.4 关键发现

**代码无并发 bug**：经过深入分析和测试验证，超时问题是 SQLite 数据库本身的并发限制，而非代码问题。

**验证证据**：
- 低并发（10 连接）测试：图片详情 QPS 8445，P99 < 10ms ✅
- 单次请求响应时间：9ms ✅
- 尝试优化 ViewBuffer 后超时反而增加，证明瓶颈在数据库层
