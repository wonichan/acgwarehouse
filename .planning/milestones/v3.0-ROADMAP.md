# Milestone v3.0: 导入后任务平台化

**Status:** ✅ SHIPPED 2026-03-29
**Archived:** 2026-04-03
**Phases:** 11-14
**Total Plans:** 15

## Overview

v3.0 将导入后的 AI 标签等后台处理能力统一收敛到批次 / 平台任务模型中，补上了“大批量导入后仍需逐张人工触发”的主路径缺口。里程碑交付包括统一任务平台、自动 AI 入队、批次优先的后台监控与控制，以及面向运营的回填恢复与失败分组诊断。

## Phases

### Phase 11: 任务平台基础与批次模型

**Goal**: 建立统一的导入批次、后处理任务与状态流转模型，为所有导入后任务提供同一平台入口。
**Depends on**: Phase 10 (v2.0 complete)
**Plans**: 4 plans

Plans:

- [x] 11-01: 定义导入批次、任务与任务状态模型
- [x] 11-02: 建立任务持久化与状态流转规则
- [x] 11-03: 接入统一调度入口与分发骨架
- [x] 11-04: 提供后台查询所需的批次 / 任务读模型

**Details:**
落地 `task_batches` / `platform_tasks` 数据模型、去重与聚合规则，并保留 `async_jobs` 作为执行层，让后续自动调度与后台监控都能在统一平台语义上继续演进。

### Phase 12: 导入后任务接入与自动调度

**Goal**: 将 AI 标签与导入后处理链路接入统一平台，实现导入完成后的自动入队与条件过滤。
**Depends on**: Phase 11
**Plans**: 4 plans

Plans:

- [x] 12-01: 数据模型变更与查询基础（`image_tag.source` 字段 + `FindImagesWithoutAITags`）
- [x] 12-02: 定时扫描服务实现（`AITagAutoScheduler` + 配置项）
- [x] 12-03: 集成调度服务到应用启动流程
- [x] 12-04: 端到端验证与真实批次测试

**Details:**
建立“缩略图已完成 + 当前无 AI 标签 + 当前无活动 AI 任务”的自动入队口径，并把周期性扫描、应用生命周期接线和 SQLite 集成验证一起补齐，形成导入后自动整理闭环。

### Phase 13: 后台监控与队列控制

**Goal**: 在后台管理页面提供按批次监控、队列状态查看和核心控制动作。
**Depends on**: Phase 12
**Plans**: 4 plans

Plans:

- [x] 13-01: 设计后台任务平台页的批次与状态视图
- [x] 13-02: 提供队列统计与任务明细接口
- [x] 13-03: 实现暂停 / 继续 / 取消 / 清空控制动作
- [x] 13-04: 实现失败任务重试与操作反馈

**Details:**
后台主入口改为 batch-first 监控台，运营侧可以看到队列 / 批次 / 任务运行态，并执行 pause、resume、cancel、clear、retry 等核心动作。失败重试被锁定为“新建批次”，避免破坏历史追踪语义。

### Phase 14: 补跑恢复与运营收尾

**Goal**: 完成“未打标签图片”批量补入队、失败隔离与恢复体验，补齐运营闭环。
**Depends on**: Phase 13
**Plans**: 3 plans

Plans:

- [x] 14-01: Define filtered backfill preview and execute contracts
- [x] 14-02: Prove per-image failure isolation and add grouped failure summaries
- [x] 14-03: Wire admin recovery UX and final operator verification

**Details:**
交付 preview-first 的补跑流程、结构化失败分组摘要和可重试提示，保证单图失败不会阻塞同批任务，管理员也能在后台直接定位失败原因并执行恢复操作。

## Milestone Summary

**Key Decisions:**

- 保留 `async_jobs` 作为执行层，同时把产品语义提升到 batch/task 模型，避免重写 worker 引擎。
- 自动入队默认只覆盖“无 AI 标签”的图片，以控制成本与重复处理风险。
- 失败重试必须创建新批次，而不是把旧失败任务重置为 ready。
- 回填恢复采用 preview-first 流程，并与重试动作分开，减少误操作与运营歧义。

**Issues Resolved:**

- 解决了导入后 AI 任务仍需逐张人工触发的问题。
- 解决了后台只能查看裸 `async_jobs`、缺少批次视角和统一控制入口的问题。
- 解决了补跑恢复时缺乏失败分组摘要和可解释反馈的问题。

**Issues Deferred:**

- 多任务类型的优先级 / 并发策略仍留待下一里程碑定义。
- 对“已有 AI 标签图片”的主动重跑 / 重生成能力仍未进入当前范围。
- 平台扩展到更多导入后任务类型与插件化处理器仍待下一阶段推进。

**Technical Debt Incurred:**

- v1.0 的 Phase 2 / 4 / 5 仍缺少独立 `VERIFICATION.md` 文档。
- Flutter 侧仍有少量 `analyze` warning / info，暂无 blocker error。
