## 补跑作用域

### 主入口默认作用域

| Option | Description | Selected |
|--------|-------------|----------|
| 当前筛选 | 沿用后台筛选 → 批量动作路径，基于明确结果集补跑 | ✓ |
| 全库未打标签 | 无筛选也能直接补跑全库 | |
| 来源/批次上下文 | 从 source root 或旧批次直接发起补跑 | |
| 每次先选择 | 点击后再选作用域 | |

**User's choice:** 当前筛选

### 无筛选时的行为

| Option | Description | Selected |
|--------|-------------|----------|
| 要求先筛选 | 没有筛选条件时不允许直接补跑 | ✓ |
| 默认全库 | 无筛选时退化为全库补跑 | |
| 弹确认选择 | 让管理员在“全库 / 先筛选”间选择 | |
| 隐藏按钮 | 没有筛选时不展示入口 | |

**User's choice:** 要求先筛选

### 是否增加恢复型快捷入口

| Option | Description | Selected |
|--------|-------------|----------|
| 批次快捷入口 | 从批次直接补跑仍未打标签的图片 | |
| 来源快捷入口 | 从 source root 直接补跑 | |
| 都不要 | 第 14 阶段不增加额外快捷入口 | ✓ |
| 两者都要 | 同时提供批次和来源快捷入口 | |

**User's choice:** 都不要

### “当前筛选结果”的覆盖范围

| Option | Description | Selected |
|--------|-------------|----------|
| 筛选命中全集 | 处理所有匹配当前筛选条件的图片 | ✓ |
| 当前页面结果 | 只处理当前页或已加载结果 | |
| 仅手动勾选 | 必须先勾选图片再补跑 | |
| 让系统判断 | 由实现阶段决定范围 | |

**User's choice:** 筛选命中全集

### 混合结果集的处理

| Option | Description | Selected |
|--------|-------------|----------|
| 自动过滤并提示 | 自动跳过已有 AI 标签图片并明确提示 | ✓ |
| 必须先筛纯净结果 | 只有结果集全是未打标签图片才能补跑 | |
| 照常执行不提示 | 后台跳过，但界面不解释 | |
| 执行前给预览 | 先展示处理 / 跳过预览再确认 | |

**User's choice:** 自动过滤并提示
**Notes:** 补跑主入口只服务当前筛选命中的完整结果集，不扩展到批次 / 来源快捷恢复入口。

## 未打标签口径

### 最终定义

| Option | Description | Selected |
|--------|-------------|----------|
| 当前无 AI 标签 | 只要当前无 AI 标签结果，就属于候选 | ✓ |
| 历史从未生成 | 只有从没生成过 AI 标签才算候选 | |
| 两种都支持 | 给管理员提供切换模式 | |
| 你来定 | 由实现阶段按代码一致性决定 | |

**User's choice:** 当前无 AI 标签

### 删除过 AI 标签的图片

| Option | Description | Selected |
|--------|-------------|----------|
| 要，重新纳入 | 当前无标签就重新进入候选 | ✓ |
| 不要再纳入 | 历史上跑过一次就排除 | |
| 给管理员选择 | 提供开关 | |
| 仅失败过才纳入 | 只对失败历史图片重新纳入 | |

**User's choice:** 要，重新纳入

### 历史失败过的图片

| Option | Description | Selected |
|--------|-------------|----------|
| 照常纳入 | 只看当前是否缺结果，不看失败历史 | ✓ |
| 默认排除 | 失败过的图片不再自动进入候选 | |
| 单独分组 | 仍纳入，但要单独标出 | |
| 只纳入失败过的 | 把补跑主要当成失败恢复入口 | |

**User's choice:** 照常纳入

### 已有在途任务的图片

| Option | Description | Selected |
|--------|-------------|----------|
| 排除并说明 | 不重复入队，并解释被跳过原因 | ✓ |
| 允许重复入队 | 即使正在跑，也允许补跑新任务 | |
| 执行前拦截 | 命中这类图片就中止整个动作 | |
| 由系统判断 | 不在产品层锁定规则 | |

**User's choice:** 排除并说明
**Notes:** “未打标签”最终按当前缺结果口径定义，且与现有自动扫描规则保持一致。

## 补跑反馈

### 执行前确认内容

| Option | Description | Selected |
|--------|-------------|----------|
| 命中+新建+跳过 | 确认前即展示三类关键数量 | ✓ |
| 只显示命中数 | 更简洁，但解释性弱 | |
| 只做风险确认 | 只提示会创建新批次 | |
| 不做确认 | 点击后直接执行 | |

**User's choice:** 命中+新建+跳过

### 执行后主反馈

| Option | Description | Selected |
|--------|-------------|----------|
| Toast+跳新批次 | 延续现有重试路径，直接进入新批次 | ✓ |
| Toast即可 | 只提示结果，不自动跳转 | |
| 停留原地 | 仅刷新原列表 | |
| 弹结果面板 | 用更重的结果摘要面板替代 toast | |

**User's choice:** Toast+跳新批次

### 跳过数的解释粒度

| Option | Description | Selected |
|--------|-------------|----------|
| 细分原因 | 至少拆成“已有 AI 标签 / 已有在途任务” | ✓ |
| 只给总数 | 不拆原因 | |
| 只在详情页细分 | toast 只给总数，详情里再看 | |
| 完全不显示 | 只显示新建数 | |

**User's choice:** 细分原因

### 0 创建时的反馈

| Option | Description | Selected |
|--------|-------------|----------|
| 明确说明原因 | 直接解释为什么没有可补跑图片 | ✓ |
| 普通空成功 | 当成普通成功处理 | |
| 当作警告 | 用 warning 风格强调无实际效果 | |
| 阻止执行 | 确认前就拦截这个空动作 | |

**User's choice:** 明确说明原因
**Notes:** 补跑反馈必须可解释、可验证，并与 Phase 13 的“新批次恢复反馈”保持一致。

## 失败摘要粒度

### 批次层默认展示方式

| Option | Description | Selected |
|--------|-------------|----------|
| 按原因聚合 | 聚合展示主要失败原因及其数量 | ✓ |
| 最新一条失败 | 只看最新错误 | |
| Top 3 原因 | 只列 2-3 类主要原因 | |
| 只显示失败数 | 不展示原因内容 | |

**User's choice:** 按原因聚合

### 聚合摘要默认信息

| Option | Description | Selected |
|--------|-------------|----------|
| 原因+数量 | 先服务运营决策 | ✓ |
| 原因+示例文件 | 附代表性文件 | |
| 原因+最后时间 | 看失败的新旧 | |
| 全部都要 | 默认堆满更多信息 | |

**User's choice:** 原因+数量

### 是否给出恢复提示

| Option | Description | Selected |
|--------|-------------|----------|
| 要，直接提示 | 在摘要里直接告诉运营是否适合重试 | ✓ |
| 不要，只给事实 | 只陈述错误和数量 | |
| 仅任务级提示 | 任务明细里再给建议 | |
| 仅明显可判定时 | 有把握时才提示 | |

**User's choice:** 要，直接提示

### 默认展示位置

| Option | Description | Selected |
|--------|-------------|----------|
| 批次列表直接显示 | 在 batch-first 列表层就能看见 | ✓ |
| 批次展开区 | 点开批次后再看 | |
| 任务明细区 | 下钻后才出现 | |
| 两处都显示 | 列表和明细都放 | |

**User's choice:** 批次列表直接显示
**Notes:** 失败摘要的目标不是完整排障，而是先帮助管理员判断“要不要救、该怎么救”。

## 恢复主动作

### 失败批次默认主动作

| Option | Description | Selected |
|--------|-------------|----------|
| 重试失败任务 | 恢复主路径继续以重试失败任务为中心 | ✓ |
| 先看失败摘要 | 默认先引导到摘要而不是动作 | |
| 补跑未打标签 | 把补跑提升成失败批次主动作 | |
| 双主动作并列 | 重试和补跑并列强调 | |

**User's choice:** 重试失败任务

### 不适合直接重试时的引导

| Option | Description | Selected |
|--------|-------------|----------|
| 保留重试+给警示 | 仍保留恢复动作，但显式提示风险 | ✓ |
| 先不显示重试 | 不适合时直接隐藏重试 | |
| 改成查看详情 | 主动作切成详情 | |
| 让系统自动决定 | 由实现层决定不同引导 | |

**User's choice:** 保留重试+给警示

### 任务级恢复入口定位

| Option | Description | Selected |
|--------|-------------|----------|
| 保留次级入口 | 批次级为主，单任务仅补充 | ✓ |
| 和批次同级 | 同样重要 | |
| 弱化隐藏 | 尽量不突出任务级恢复 | |
| 你来定 | 由实现阶段决定 | |

**User's choice:** 保留次级入口

### 补跑入口层级

| Option | Description | Selected |
|--------|-------------|----------|
| 全局/筛选工具栏 | 作为独立运营入口，不混入失败批次动作 | ✓ |
| 批次行操作 | 主要从批次列表发起 | |
| 任务明细区 | 选中批次后再补跑 | |
| 到时候再定 | 暂不锁层级 | |

**User's choice:** 全局/筛选工具栏
**Notes:** “补跑”和“重试失败任务”是两条不同心智路径，前者面向当前缺结果，后者面向已有失败任务恢复。
