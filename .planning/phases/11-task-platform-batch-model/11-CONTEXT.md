# Phase 11: 任务平台基础与批次模型 - Context

**Gathered:** 2026-03-24
**Status:** Ready for planning

<domain>
## Phase Boundary

本阶段交付导入后任务平台的基础批次模型、统一任务模型、状态流转语义，以及供后台查询使用的批次 / 任务读模型。

本阶段聚焦“把导入后的后台处理纳入同一平台入口并可追踪”，不在这里落地 AI 自动入队规则（Phase 12）、后台控制动作与完整监控页面（Phase 13），也不扩张为全站所有异步任务的一次性总重构。

</domain>

<decisions>
## Implementation Decisions

### 平台覆盖范围
- Phase 11 的任务平台从第一版开始覆盖三类来源：导入触发、单图手动触发、批量补跑。
- 单图 AI 手动触发也接入同一任务平台，不再长期保留平台外的平行语义。
- 单图手动触发形成自己的小批次；即使只有 1 张图，也按批次追踪。
- 缩略图任务作为导入后任务平台中的正式可见任务保留，不隐藏为纯内部步骤。
- `image_imported` 作为平台内部事件 / 调度起点保留，但不作为用户重点关注的独立可见任务主体。
- 未来补跑批次与导入批次共享同一批次模型和同一套任务类型定义，区别主要体现在批次来源类型上。

### 批次定义
- 一个批次的自然单位是“一次触发动作”：一次导入、一次单图手动触发、一次批量补跑，各自形成一个批次。
- 一次扫描动作即使覆盖多个 scan root，也仍然属于同一个批次；批次内记录多个来源根目录。
- 文件夹监控模式下，连续发现的新文件应按短窗口合批，但窗口取值以“适中平衡”为目标，避免既碎成大量 1 图批次，也避免把相隔较远的导入混成一批。
- 用户再次对同一来源执行手动重扫时，应新建新的批次，而不是追加到旧批次。
- 已存在 / 未变更图片需要计入批次摘要，但不为其创建新的后处理任务。
- 单图手动触发形成的小批次与普通导入批次在后台列表中并列展示，仅通过来源类型区分。
- 批次标识以系统自动生成摘要为主，包含来源类型、时间、来源摘要、数量等信息，不依赖人工命名。

### 状态口径
- 批次总状态需要支持“部分失败 / 部分完成”语义，而不是只能二选一地看成成功或失败。
- Phase 11 的基础模型就应预留 `cancelled` 与 `skipped` 这类终态，避免 Phase 13 再次改动核心状态模型。
- 批次完成的判定口径是：纳入该批次的后处理任务全部进入终态后，批次才算完成；不能在“导入动作结束”时就提前判定完成。
- 某些任务失败时，其它任务继续推进；平台默认采用失败隔离而不是整批停下。

### 重复触发保护
- “未变更图片不重复入队”以图片内容 / 版本未变为主判断，而不是仅靠路径相同判断。
- 当用户在前一批尚未完全结束时再次触发同源处理，可以创建新批次保留新的操作记录，但对已在跑或已完成且未变更的任务不再重复入队。
- 重复保护按任务类型生效：同一张图若缩略图已完成但 AI 标签未完成，则只补缺失的任务类型，不整图全跳过，也不整图全重跑。
- 因“未变更 / 重复”而被跳过的图片或任务，不需要逐条展开为任务记录，但必须在批次摘要里体现跳过数量与原因摘要。

### OpenCode's Discretion
- 文件夹监控“短窗口合批”的具体时间阈值与合并规则由 OpenCode 在规划 / 实施阶段细化。
- 批次来源类型、批次系统摘要的具体命名格式、显示文案与排序细节由 OpenCode 决定。
- `skipped` 是落为任务终态、批次成员属性，还是仅作为批次汇总统计的一部分，由 OpenCode 结合读模型和状态机一致性做最终设计。
- 批次摘要中的“未变更 / 重复 / 已补缺”等统计是持久化存储还是查询时聚合，由 OpenCode 结合 SQLite 读写复杂度决定。

</decisions>

<specifics>
## Specific Ideas

- 这次平台化不是“给 AI 标签补一个批量入口”，而是要从 Phase 11 开始建立统一的导入后任务语义。
- 单图手动触发也应该看起来像平台任务，而不是继续留在平台之外做特例。
- 后台对人真正有意义的是导入后任务本身（缩略图、AI 标签等），不是 `image_imported` 这种内部事件节点。
- 新批次可以继续创建，但对未变更内容和已完成任务要自动补缺而不是重复轰炸队列，这样才能同时保留历史与幂等性。

</specifics>

<code_context>
## Existing Code Insights

### Reusable Assets
- `internal/worker/job_manager.go`: 已有统一任务队列、worker pool、Pause/Resume、任务持久化入队和 `ready/running/finished/failed` 基础状态流转，可作为平台执行底座。
- `internal/domain/async_job.go`: 已有统一任务实体，可在此基础上扩展批次关联和更完整的生命周期字段。
- `internal/service/scanner_service.go`: 当前导入链路已经在保存新图片后创建 `image_imported` 任务，是批次创建与导入后任务挂接的直接入口。
- `internal/app/bootstrap.go`: 已有 `image_imported -> thumbnail_generate` 的调度骨架，以及 AI handler 注册点，可复用为平台化后的统一调度入口。
- `internal/repository/schema.go` 与 `internal/repository/job_repository.go`: 已有 `async_jobs` 表与任务查询模式，可扩展为批次表、批次关联及读模型查询。
- `internal/service/admin_service.go`、`internal/handler/admin_handler.go`、`internal/handler/routes.go`: 已有后台 summary / jobs 接口与管理后台路由，可继续扩展批次 / 任务读模型查询。

### Established Patterns
- 项目当前主路径锁定为 Go 后端 + SQLite 单机部署；Phase 11 不引入分布式队列或多机调度。
- 导入链路已经遵循“只有新图片才创建后续任务”的基本去重思路；Phase 11 需要把这条规则提升为批次 / 任务级平台语义，而不是只停留在 `SaveImage()` 插入层。
- 现有后台接口采用统一 JSON 返回与 `/admin/api/*` 路由模式，后续批次 / 任务查询应延续这一风格。
- v3.0 已明确 AI 标签是首个重点任务类型，但自动入队条件本身放在 Phase 12 再实现。

### Integration Points
- 批次模型需要接入 `ScannerService`，在一次导入触发开始时创建批次，并在任务创建与跳过统计时回写批次上下文。
- 平台任务模型需要承接当前 `image_imported`、`thumbnail_generate` 与手动 `ai_tag_generation` 的语义，并为未来批量补跑复用。
- 读模型需要服务后续后台查询，因此批次级统计、来源信息、任务类型计数、跳过摘要等应在 Phase 11 就被纳入设计。
- 需要避免旧异步入口与新平台入口长期双轨并存；现有手动单图 AI 入口应朝“同平台、不同来源批次”收敛。

</code_context>

<deferred>
## Deferred Ideas

- AI 标签“默认仅无 AI 标签图片自动入队”的具体规则与挂接时机 —— Phase 12。
- 后台暂停 / 继续 / 取消 / 清空 / 重试等控制动作与完整监控 UI —— Phase 13。
- 失败恢复体验、未打标签图片批量补入队的运营闭环细节 —— Phase 14。
- 把 manual_scan、FTS rebuild 等全站所有异步任务都纳入同一平台 —— 当前不作为 Phase 11 第一版范围。

</deferred>
