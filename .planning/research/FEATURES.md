# 特性研究：v4.0 Windows Photos 风格重构与计算侧车

**领域:** 桌面图片库管理系统（ACG 图库）
**里程碑:** v4.0 Windows Photos 风格重构与计算侧车
**调研时间:** 2026-04-03
**信心等级:** MEDIUM（基于现有产品调研、Windows Photos 参考文档、行业标准分析）

## Table Stakes（用户期望具备）

这些特性用户默认认为应该存在。缺失会导致产品感觉不完整。

| 特性 | 为何期望 | 复杂度 | 备注 |
|------|----------|--------|------|
| **侧边导航栏** | Windows Photos 标准范式，用户习惯从侧边栏切换视图（收藏、文件夹、OneDrive） | 中 | 需重构现有 Flutter NavigationView，采用可收起设计（展开 240px / 收起 48px），支持收藏夹、文件夹、导入任务监控入口 |
| **顶部工具栏** | 桌面应用标准交互模式，快速访问搜索、导入、设置等核心操作 | 低 | 固定高度 52px，包含搜索栏、导入按钮、设置入口。需与侧边导航协调布局 |
| **瀑布流/网格视图切换** | Windows Photos 提供 River/Square 模式，用户期望浏览方式可选 | 中 | 瀑布流适合艺术作品（保留原始宽高比），方块模式适合快速浏览大量图片。需适配虚拟滚动与缩略图缓存策略 |
| **独立查看器窗口** | Windows Photos 双击图片在新窗口打开，不阻塞主图库操作 | 高 | 非模态设计，支持同时打开多个查看器对比图片。需 Flutter 窗口管理器支持，进程间通信通过 Go API |
| **底部胶片条（Filmstrip）** | 查看时可快速切换同文件夹其他图片，Windows Photos 标准体验 | 中 | 显示当前文件夹缩略图序列，支持左右滑动、点击跳转、Ctrl/Shift 多选。需与查看器独立窗口协调状态同步 |
| **图片缩略图虚拟滚动** | 大图库（10k+ 图片）流畅浏览的必需能力 | 中 | 现有 Flutter 已具备，需重构适配新布局，优化缓存策略（优先加载可见区域，预加载下一屏） |
| **搜索栏（文件名/标签）** | 图片库管理的核心交互，快速定位目标图片 | 低 | 现有 Go + SQLite FTS5 已支持，需重构前端 UI 集成到顶部工具栏，提供实时搜索建议 |
| **图片导入流程** | 用户期望从本地文件夹导入图片到图库 | 低 | 现有 Go 扫描服务已支持，需重构前端导入入口（拖拽导入、文件夹选择），提供进度反馈 |
| **Python 计算侧车进程管理** | 用户期望后台任务（重复检测、哈希计算）可靠运行，不崩溃、不卡死 | 高 | Go 负责启动、健康检查、优雅关闭 Python 进程。需实现心跳检测（超时自动重启）、端口冲突避免（随机端口）、僵尸进程清理 |

## Differentiators（竞争优势）

这些特性让产品脱颖而出。非必需，但有价值。

| 特性 | 价值主张 | 复杂度 | 备注 |
|------|----------|--------|------|
| **智能重复检测（精确 + 感知哈希）** | 解决"相同图片不同格式/分辨率"的真正痛点，用户无需手动逐张对比 | 高 | Python 侧车承接：SHA256 精确重复 + pHash/dHash 感知哈希 + Union-Find 聚类分组。Go 编排批次管理、结果落库。参考 Tidy 应用：区分"完全相同" vs "相似照片（同一时刻不同拍摄）"，建议保留策略（分辨率最高/清晰度最佳） |
| **相似图片分组与推荐保留** | 从"检测重复"升级到"推荐最佳版本"，减少用户决策成本 | 高 | Python 计算层提供质量评估信号（锐度、曝光、EXIF 完整度）。Go 返回推荐保留图片 ID + 理由。前端展示分组对比界面，用户可接受推荐或手动选择 |
| **AI 标签与导入后任务平台集成** | 导入完成自动触发 AI 标签入队，后台可监控、可控制、可恢复 | 中 | v3.0 已完成平台基础，v4.0 需重构前端监控入口集成到侧边导航，提供批次优先视图 + 任务明细下钻 |
| **侧边导航"导入任务监控"入口** | 将后台运营工具收敛到桌面主体验，用户无需切换到 Web Admin | 中 | 新增侧边栏入口：导入批次列表、任务进度、失败摘要、重试/取消/清空操作。需与现有 Go 任务平台 API 集成 |
| **Python 侧车故障诊断与回退** | 用户/管理员可查看侧车健康状态、错误日志、手动重启 | 高 | 前端提供"系统状态"入口，显示 Python 进程状态（运行中/已停止/重启中）、最近错误摘要、手动重启按钮。Go 负责进程监控 + 错误日志暴露 |
| **桌面窗口打包为单文件分发** | 用户无需安装 Python/Go，解压即用，降低部署门槛 | 高 | Flutter 打包 Windows Release + Go 编译为 exe + Python PyInstaller 单文件 → 合并为 zip/installer。需设计启动顺序（Flutter 拉起 Go → Go 拉起 Python） |
| **侧边栏文件夹树形视图** | Windows Photos 提供文件夹导航，ACG 用户习惯按目录浏览 | 中 | 左侧导航显示文件夹层级（Pictures + 用户添加的其他文件夹），点击切换图库视图。需与 Go 文件扫描服务集成，实时反映目录结构 |

## Anti-Features（常被请求但易引发问题）

这些特性看似有用但会带来麻烦。

| 特性 | 为何被请求 | 为何有问题 | 替代方案 |
|------|------------|------------|----------|
| **实时全库重复检测** | "我想一键检测所有重复，立刻看到结果" | 10k+ 图片全库对比计算量巨大（O(n²)），阻塞 UI 数小时，用户无法取消中途离开 | **触发式分批检测：** 用户点击"开始检测" → Go 创建批次 → Python 按批次计算 → WebSocket 推送进度 → 前端显示进度条 + 可暂停/取消 |
| **Python 直接持久化到数据库** | "Python 计算完直接写 SQLite，减少 Go 中转开销" | 违反架构边界（Python 只负责计算，不应持有业务状态），导致职责混乱、难以调试、Go 无法审计计算结果 | **Go 编排落库：** Python 返回计算结果（JSON） → Go 验证、清洗、写入 duplicate_groups 表 → 保留审计日志 |
| **全局实时搜索（全字段模糊匹配）** | "我想搜任意关键词都能立刻匹配" | SQLite FTS5 全字段模糊搜索性能差（无索引优化），10k+ 图片库搜索延迟超 500ms，拖慢 UI | **分字段搜索建议：** 文件名精确匹配 + 标签 AND 语义 + 提供"仅搜索文件名" / "仅搜索标签"切换选项 |
| **查看器窗口模态锁定主图库** | "打开图片后主窗口不应操作，防止用户混淆" | 阻塞用户多任务流程（对比多张图片时无法同时浏览其他文件夹），违反 Windows Photos 非模态设计原则 | **独立非模态窗口：** 双击图片打开独立查看器，主图库窗口可继续浏览、搜索、导入。支持同时打开多个查看器 |
| **硬编码 Python 服务端口（8080/5000）** | "固定端口方便调试，不用每次动态获取" | 端口冲突风险高（用户机器可能运行其他服务），启动失败难以诊断，违反进程管理最佳实践 | **随机端口 + stdin/stdout 通信：** Python 监听 `127.0.0.1:0`（随机端口） → 打印端口到 stdout → Go 读取并建立 HTTP 连接 |
| **图片编辑功能（裁剪/滤镜/调色）** | "Windows Photos 有编辑功能，我们也应该有" | v4.0 里程碑聚焦浏览/查看/计算重构，编辑是独立大型特性，引入会稀释核心目标 | **延后到 v5.0+：** v4.0 完成桌面 UX + Python 侧车地基后，后续里程碑可引入图片编辑（裁剪、滤镜、AI 背景移除） |

### 依赖说明

- **独立查看器窗口 requires Flutter 窗口管理器：** 需 `window_manager` + `flutter_acrylic` 插件支持多窗口创建与管理
- **独立查看器窗口 requires 底部胶片条状态同步：** 查看器窗口需知道当前文件夹图片列表，通过 Go WebSocket 推送同步
- **智能重复检测 requires Python 计算侧车进程管理：** Python 侧车必须先启动并稳定运行，才能承接计算任务
- **Python 计算侧车进程管理 requires Go 进程启动/监控：** Go 作为主控负责拉起 Python 进程、监听 stdout 获取端口、建立 HTTP 连接、心跳检测
- **Python 计算侧车进程管理 requires Go ↔ Python HTTP 通信契约：** 需定义统一 REST API 契约（POST /compute/duplicates → 返回 JSON）
- **相似图片推荐保留 enhances Python 质量评估信号：** 基础重复检测后，Python 可扩展计算锐度/EXIF 完整度，为推荐策略提供依据
- **桌面窗口打包 requires 启动顺序编排：** Flutter 先启动 → 拉起 Go exe → Go 拉起 Python exe → Go 等待 Python 端口 → Go 启动 HTTP 服务 → Flutter 连接 Go API
- **Python 侧车故障诊断 enhances Go 错误日志暴露 API：** 进程管理基础上，Go 需额外提供 `/system/python-status` API，暴露进程状态与最近错误
- **瀑布流/网格视图切换 requires 虚拟滚动优化：** 瀑布流需特殊虚拟滚动策略（按行加载而非按固定高度分页），方块模式可沿用现有分页虚拟滚动

## MVP 定义（v4.0 里程碑）

### 本里程碑必需（Launch With）

验证桌面 UX 重构 + Python 侧车可行性所需的最小可行特性集。

- [x] **侧边导航栏基础框架** — 桌面 UX 重构的起点，提供收藏、文件夹、导入任务入口
- [x] **顶部工具栏（搜索 + 导入 + 设置）** — 核心交互入口，集成现有 Go API
- [x] **瀑布流视图** — 验证虚拟滚动 + 缩略图缓存在新布局下的性能
- [x] **Python 计算侧车进程管理** — 架构拆分的核心验证，Go ↔ Python 生命周期必须稳定
- [x] **智能重复检测（精确 + 感知哈希）** — Python 侧车的首个计算能力验证，承接现有 Go 重复检测逻辑
- [x] **侧边导航"导入任务监控"入口** — 将 v3.0 任务平台收敛到桌面主体验
- [x] **Go ↔ Python HTTP 通信契约** — 计算侧车的基础通信协议，后续所有计算任务都依赖此契约
- [x] **桌面窗口打包基础验证** — Windows 单文件分发可行性验证（Flutter + Go + Python 合并）

### 验证后添加（v4.x 迭代）

核心骨架验证后，逐步补齐的增强特性。

- [ ] **独立查看器窗口** — 验证桌面 UX 骨架后，引入非模态查看器体验（需窗口管理器额外开发）
- [ ] **底部胶片条（Filmstrip）** — 独立查看器窗口的配套体验，支持快速切换同文件夹图片
- [ ] **相似图片推荐保留** — 基础重复检测验证后，扩展质量评估与推荐策略
- [ ] **Python 侧车故障诊断** — 进程管理验证后，暴露健康状态与错误日志给用户/管理员
- [ ] **侧边栏文件夹树形视图** — 验证侧边导航框架后，扩展文件夹层级导航
- [ ] **方块网格视图模式** — 瀑布流验证后，增加视图切换选项

### 后续里程碑考虑（v5.0+）

产品成熟后再引入的扩展特性。

- [ ] **图片编辑功能（裁剪/滤镜/调色）** — v4.0 完成浏览/查看重构后，后续引入编辑能力
- [ ] **AI 背景移除/替换** — 需要 Python 侧车扩展深度学习模型，计算资源与打包体积显著增加
- [ ] **OCR 文字识别** — Python 侧车扩展 OCR 能力（Tesseract/EasyOCR），需评估中文 ACG 图片场景适用性
- [ ] **Linux 桌面端支持** — Windows 主路径验证后，评估跨端复制到 Linux
- [ ] **云同步（OneDrive/iCloud 集成）** — 本地部署模型验证后，扩展云端同步能力
